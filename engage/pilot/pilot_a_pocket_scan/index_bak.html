Pilot A — Pocket Scan

Prove pockets exist. Locate them. Verify repeatability once.

This is the fastest path to turning “the room average is fine” into a spatial truth: where RH/temp/VPD pockets live inside the canopy, when they appear, and whether they repeat on comparable cycles.

Platform status / scope note: This pilot page reflects the current Pilot A operating approach and report format. Outputs, UI screens, terminology, and acceptance tests may evolve; any engagement is governed by the statement of work and acceptance criteria agreed for your site.

Why this matters

  • Dense canopies + imperfect airflow create local humidity pockets that sensors on walls can’t represent. Crowding and poor air circulation are repeatedly cited as contributors to disease-favorable microclimates.
  • PM and botrytis are strongly influenced by humidity dynamics and airflow; pockets are a common root cause, not a rare edge case.
  • Pockets are spatial and cyclical. Averages can look “fine” while worst-case zones stay wet long enough to create risk.

Best fit

You’re seeing

  • “Mystery corners” / under-canopy wet strips
  • Night spikes / dehu cycling weirdness
  • Pressure that doesn’t match the dashboard
  • PM/botrytis risk that feels inconsistent with averages

You want

  • Proof pockets exist before bigger spend
  • A map you can act on (not just plots)
  • One verification pass on a comparable cycle

Deliverables

  • Zone map: pockets + gradients + “risk surfaces” (where it stays wet)
  • 3-slice stack: top / mid / under-canopy slices (layer effects vs room-wide effects)
  • Trigger notes: what coincides with pockets (lights/irrigation/dehu/HVAC cycles)
  • Repeatability overlay: two comparable days aligned to the same 24-hour axis
  • Validity windows: confident vs abstains (and why)
  • Receipts: device IDs, firmware/software versions, calibration/config state
  • Readout: top findings + likely failure modes + recommended next steps

How to read the deliverables

Zone map: pockets + gradients + risk surface

Use this to answer: Where is it consistently worse? The risk surface shows gradients; pocket markers name specific zones you can walk to (corners, under-canopy strips, behind racks).

Action pattern: start with the top 1–2 pockets by severity, confirm the physical cause (air path, obstructions, dehu throw), then apply one targeted change at a time.

3-slice stack (top / mid / under-canopy)

Use this to answer: Is it a canopy-layer effect or a room-wide effect? If under-canopy is consistently worse than top canopy, airflow and canopy density are usually implicated.

Interpretation cue: layer separation helps you choose the right intervention class (airflow path vs setpoints vs canopy structure).

Trigger notes: event timeline + pocket severity

Use this to answer: What cycles correlate with pocket severity? Severity peaks are aligned with lights/irrigation/dehu/HVAC bands, alongside a validity ribbon that marks where interpretation is earned.

Practical use: focus interventions on the event windows that consistently precede severity spikes (not on averages across the entire day).

Repeatability overlay: two comparable days

Use this to answer: Does the pocket repeat on comparable cycles? Two days are aligned to the same 24-hour axis. Similar timing and rank strengthens confidence; divergence is flagged.

Decision rule: “repeatable once” means pocket location and severity ordering stay broadly stable when cycles are truly comparable.

Validity windows: confident vs abstain

Operators shouldn’t have to guess when comparisons are valid. We explicitly label when the system is CONFIDENT and when it ABSTAINS (and why).

Validity ribbon (with reasons)

Use this to answer: When should I trust the comparison? Why not? ABSTAIN segments are annotated with concrete reasons so you know what to fix (cycle mismatch, coverage gap, calibration stale, etc.).

Operator meaning: ABSTAIN doesn’t mean “nothing happened” — it means “don’t interpret this as verified improvement (or cause) until criteria are met.”

Public criteria (why we abstain)

Interpretation is earned: clock sync, calibration status, coverage, and cycle comparability determine whether a comparison is valid.

Practical takeaway: when criteria aren’t met, we still report observations — but we do not over-claim causality or efficacy.

Receipts: provenance you can hand to anyone

When changes cost money (labor, downtime, equipment), you need receipts: what ran, on which devices, with which firmware/config, and whether acceptance tests passed.

Run manifest (“receipt”)

Use this to answer: What exactly was measured, with what versions, and under what configuration? This is the audit-style record you can hand to facilities, ownership, vendors, or compliance.

Why it matters: if a result is disputed (“that wasn’t the same setup”), the manifest is the proof of the run state and the criteria it passed.

Rig topology snapshot

Use this to answer: How was the rig laid out, and which nodes produced the data? A topology snapshot captures the measurement graph at run start: node IDs, versions, and captured config state.

Practical use: if a zone looks suspicious, topology helps trace whether it’s a real spatial effect or a coverage/layout artifact.

Typical timeline

  1. Site call (30–45 min): room geometry, canopy stage, constraints, cycles.
  2. Install + commissioning: acceptance tests → baseline capture.
  3. Verification pass: repeat on a comparable day/cycle (prove repeatability).
  4. Readout: zone map + pocket list + action recommendations + receipts.

Readout (operator-ready)

Top findings one-pager

Use this to answer: What should we do next? A single page that condenses the pilot into findings, likely failure modes, and recommended next steps (with one key result you can repeat in a meeting).

Practical use: send this to stakeholders who don’t want the full data pack, then attach receipts for credibility.

Ranked pocket cards

Use this to answer: Where do we walk first, and what do we change? Pocket cards translate the map into operator actions: location, severity, trigger association, and recommended intervention class.

Decision pattern: fix the top pocket, re-run the verification pass, and only then move to the next pocket.

Acceptance tests (examples)

  • Validity uptime: system reports when measurement is trustworthy vs not.
  • Repeatability: pocket location and ranking remain stable on comparable cycles.
  • Actionability: map points to specific zones (corners, under-canopy strips, behind racks).

What we install (high level)

  • Portable measurement rig (non-permanent)
  • Commissioning checklist + labeling + run configuration
  • Portal access to validity + receipts + maps

Exact hardware layout depends on room size, canopy density, and airflow paths.

Operator FAQ (practical)

What if the dashboard average looks fine?

That’s the point. Pockets are spatial and cyclical. Averages can hide worst-case zones.

What does “ABSTAIN” mean for me?

It means we won’t claim an improvement (or causal explanation) for that segment. You still get observations, but the system is telling you: do not interpret this as verified.

What changes do you usually recommend?

Targeted, zone-specific interventions: airflow path corrections, dehu placement/strategy adjustments, canopy density/defoliation considerations, and rack/obstruction changes. The readout ties recommendations directly to pocket zones and triggers.

How do I use this with my team?

Use the pocket cards / zone map for “walk-to-this-spot” actions, and use receipts for handoff to facilities/engineering or vendors.
