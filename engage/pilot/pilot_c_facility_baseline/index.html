Why facilities buy this

  • The same airflow and humidity dynamics that drive PM/botrytis risk can vary room-to-room—even under the “same SOP.”
  • Standardization reduces regressions when staff change, SOPs drift, equipment is swapped, or rooms get rebalanced.
  • The facility problem in one sentence: you don’t have “a grow”—you have multiple microclimate systems. Pilot C stops the cycle of re-learning every room, every time.

Primary outcome

  • Variance report: why Room 2 behaves like a different planet (where/when it diverges, and plausible drivers)
  • Templates: placement + acceptance tests that travel room-to-room (so “baseline” is repeatable, not artisanal)
  • Rollout playbook: how to baseline new rooms without guessing (stepwise, auditable, scalable)

Deliverables

  • Per-room zone maps: pockets + gradients + cycle timing
  • Cross-room comparisons: common pocket archetypes and outliers
  • Facility baseline package: configs, receipts, validity uptime, change history
  • Operator-ready standard: what “good” looks like and how to verify it
  • Templates that travel: placement + acceptance tests that transfer
  • Rollout playbook: baseline new rooms without guessing

How to read the deliverables

Per-room zone maps (2–4 rooms)

Use this to answer: What does each room actually do across the cycle? Each room gets a zone map with pockets, gradients, and cycle timing cues so hotspots can be aligned to triggers (lights/irrigation/night/dehu).

Practical use: this is how you stop arguing about “that room feels weird” and start pointing to repeatable zones by location and timing.

Cross-room comparisons (archetypes + outliers)

Use this to answer: Which failure geometries repeat across rooms—and which room is the outlier? This turns “SOP drift” into pattern recognition: common pocket archetypes vs rooms behaving differently.

Operator payoff: once you know what repeats, you can standardize fixes; once you know the outlier, you can isolate why it deviates.

Variance report (why one room behaves differently)

Use this to answer: What’s the measurable delta between rooms—and what factors plausibly drive it? The goal is diagnostic and evidence-based (not blame-based): where/when the room diverges and what constraints likely matter.

Practical use: this is what you hand to facilities/engineering to justify changes, instead of “we think Room 2 is cursed.”

Facility baseline package (what “good” was)

Use this to answer: What exactly was the “good state” configuration, and can we reproduce it? The baseline package captures configs, receipts, validity uptime, and change history so “good” can travel across time and rooms.

Operator meaning: when someone says “nothing changed,” the package is the record of what actually changed—and what stayed the same.

Operator-ready standard (define “good”)

Use this to answer: What does “good” look like here, measurably—and how do we verify it? This replaces tribal knowledge with a facility-specific definition that can be checked on demand.

Practical payoff: staff changes stop causing regressions because “good” is defined and verifiable—not remembered.

Templates that travel (placement + acceptance tests)

Use this to answer: Can we apply the same placement and acceptance tests to a new room? Templates act like stencils: repeatable placement logic plus commissioning checks that transfer room-to-room.

Operator meaning: you don’t redesign measurement every time a room changes—you apply the template and verify the same criteria.

Rollout playbook (baseline new rooms without guessing)

Use this to answer: What are the steps to baseline additional rooms reliably? The playbook makes expansion repeatable: plan → commission → capture triggers → compare → standardize.

Practical payoff: you can add rooms (or reset rooms) without re-learning the whole facility from scratch.

Acceptance tests (facility-level)

Use this to answer: When is cross-room comparison valid—and when do we abstain? We only compare rooms when validity criteria are met; otherwise the system abstains (so you don’t standardize on bad evidence).

Operator meaning: “compare only when comparable” becomes enforced behavior, not a guideline people forget under pressure.

Typical timeline

  1. Plan: select 2–4 rooms + define cycles to capture (lights/irrigation/night)
  2. Baseline: instrument each room + run commissioning acceptance tests
  3. Compare: quantify variance and identify repeatable failure geometries
  4. Standardize: ship templates + rollout playbook

Acceptance tests (examples)

  • Consistency: facility “good state” is measurable, not tribal knowledge—defined by a standard you can verify.
  • Comparability: cross-room comparisons only when validity is met (otherwise abstain)—so you don’t standardize on noise.
  • Transferability: placement and acceptance tests can be applied to a new room—so expansion doesn’t restart learning.
