Operator Jobs To Be Done

We don’t build features from opinions. We build them from operator jobs: the outcomes you’re judged on, the failure modes you’re trying to avoid, and the proof you need when things get tense.

What operators actually hire this system to do

Jobs → outcomes → features (no fluff).

“Jobs To Be Done” is just a clean way to say something operators already know: you’re not buying sensors because you love data. You’re buying control. You want fewer surprises, fewer recurring problems, and fewer meetings that start with “how did this happen if we held setpoints?”

When we plan pilots and decide what to ship, we start here: what job is the operator hiring this for? Then we build the smallest set of features that reliably completes that job in real rooms.

Primary Job

“Help me run rooms I can trust — across the cycles that actually cause failures — so I’m not gambling a harvest on a calm-looking dashboard.”

When operators “hire” a system like this

  • Recurring PM/Botrytis pressure that doesn’t match the dashboard story.
  • A “mystery room” where the same SOP gives different outcomes.
  • Late-flower anxiety (nights feel risky, RH behavior feels unpredictable).
  • Compliance pain (failed lots, hold/retest cycles, remediation burn).
  • Expansion (new rooms, new facility) and they want repeatability, not folklore.

When they “fire” a system

  • It produces more dashboards but not more confidence.
  • It can’t explain where the problem is, when it triggers, or whether it repeats.
  • It can’t prove a change worked — it just says “numbers changed.”
  • It creates operational overhead without reducing failure risk.

Top Operator Jobs (the ones that create budget)

These are written in operator language on purpose. They’re the real reasons someone signs off.

  1. Find where the room lies.
    “Show me the pockets my wall sensor can’t see — corners, under-canopy, behind racks, wall-adjacent lanes.”
  2. Catch the trigger window.
    “Tell me when it happens — lights-off, night RH rise, dehu pinned, post-irrigation, HVAC staging.”
  3. Prove it repeats.
    “Separate ‘weird day’ from ‘this is how the room behaves’ so I can stop arguing and start fixing.”
  4. Verify the fix.
    “After changes, prove the pocket shrank/moved/died. Don’t make me guess and hope.”
  5. Make it defensible.
    “Give me proof I can show to leadership/QA without it turning into a faith argument.”

Desired outcomes (what “success” feels like)

  • Fewer surprises in late flower: nights feel controlled, not suspenseful.
  • Less rework: fewer emergency defoliation scrambles, fewer reactive remediation cycles.
  • Higher repeatability: rooms behave similarly under the same playbook.
  • Clear priorities: you know where to aim airflow/dehu effort first.
  • Proof of change: “we fixed it” becomes a measurable statement.

Anxieties (what operators are trying to avoid)

  • The calm dashboard followed by the ugly canopy reality.
  • The slow bleed: yield loss and quality drift that never gets pinned to a cause.
  • The compliance ambush: failed lots, holds, retests, reputational hits.
  • The leadership meeting: “how did this happen if we were in spec?”
  • The endless tuning loop: chasing setpoints that aren’t the real problem.

Feature mapping

This is how jobs become shipped product. If a feature doesn’t complete a job, it doesn’t survive roadmap review.

Operator job What the operator needs Features we build
Find where the room lies A zone-level view (not one number). Clear “bad zones” with location context. Zone map output; pocket detection; “worst zones” ranking; room overlays / simple geometry anchors.
Catch the trigger window Timing: when it happens (lights-off, night, post-irrigation, dehu pinned, HVAC staging). Cycle capture; event tagging; “trigger window” summaries; night-cycle emphasis views.
Prove it repeats Recurrence: same pocket shows up again under same conditions. Recurrence scoring; “repeat vs one-off” classification; week-over-week pocket tracking.
Verify the fix Before/after proof: pocket shrank/moved/died after intervention. A/B runs; intervention receipts; “did it work?” deltas; repeatable verification windows.
Make it defensible A narrative artifact: where/when/how often/how bad/what changed — suitable for leadership + QA. Operator-readable reports; evidence bundle export; plain-language summaries; consistent terminology.

How this connects to pilot duration

Duration isn’t a billing preference. It’s required to complete the jobs above: short pilots can find + locate, but long pilots prove recurrence + triggers, and verification requires before/after cycles. If the system can’t see cycles, it can’t do the job operators hire it for.
